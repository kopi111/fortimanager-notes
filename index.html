What FortiManager Is

A centralized management plane for Fortinet devices. Instead of logging into hundreds of FortiGates one by one, you point them at a FortiManager and operate them from a single console.

Core capabilities I keep coming back to:

  • Centralized policy and object provisioning across many devices
  • Configuration revision control (snapshots and reverts)
  • Local FortiGuard distribution (private FDS) for closed networks
  • Firmware management for managed devices
  • Mesh / star IPsec VPN deployment
  • Logging and reporting (with FortiAnalyzer features or a dedicated FortiAnalyzer)
  • Management Extension Apps (MEAs) — licensed apps that run as containers on FortiManager

Inside FortiManager — Modules & Panes

The web GUI is organized into panes that map to modules under the hood. The default ones I use most:

  • Device Manager — add/authorize devices, edit device DB, run installs
  • Policy & Objects — central firewall policies, profiles, addresses, schedules
  • VPN Manager — IPsec / SSL VPN central config
  • AP Manager / FortiSwitch Manager / Extender Manager — manage devices controlled by FortiGates
  • Fabric View — Security Fabric ratings & connectors
  • FortiGuard — local FDS / signature distribution
  • System Settings — admins, ADOMs, HA, network, advanced

Optional panes appear when FortiAnalyzer features are enabled or when a FortiAnalyzer is added: FortiView, Log View, Incidents & Events, Reports.

Administrative Domains (ADOMs)

ADOMs let you logically group devices and scope administrators to those groups. A device belongs to one ADOM; an admin can be tied to one or more ADOMs.

ADOM device modeBehavior
NormalThe whole FortiGate (all VDOMs) lives in one ADOM.
AdvancedEach VDOM can be assigned to a different ADOM.

Accounts with the Super_User profile (e.g. the default admin) can manage every ADOM. Other profiles are scoped.

FortiAnalyzer Integration

Two ways to handle logs from managed devices:

  • Add FortiAnalyzer to FortiManager — recommended for production / high log volume. FortiAnalyzer stores logs; FortiManager gets read access to its panes. Caveat: can’t enable advanced ADOM mode while a FortiAnalyzer is attached.
  • Enable FortiAnalyzer features manually on FortiManager — turns FortiManager itself into the log/report device. Capped at ~150 logs/sec, license is non-stackable, and FortiManager HA is unsupported in that mode. Fine for labs, not for real volume.

Initial Configuration & Ports

Factory defaults to know:

  • Management interface: port1 at 192.168.1.99/24
  • HTTPS and SSH enabled by default
  • Username admin, blank password (forced change on first login)

Key TCP/UDP ports (for firewall planning when FortiManager is behind a perimeter):

DirectionPurposePorts
In/OutFGFM (management to FortiGates)TCP 541 (IPv4), TCP 542 (IPv6)
InHTTPS / HTTP / SSH GUI & CLITCP 443 / 80 / 22
InWeb filter, AV, IPS updates (when acting as local FDS)UDP 53, UDP/TCP 8888, TCP 80
OutAV/IPS push updatesUDP 9443

Security baseline: deploy behind a firewall, use HTTPS / SSH only, configure trusted hosts to limit login source IPs, enable a password policy, and watch the event log.

First-Login Setup Wizard

On first login, the FortiManager Setup wizard runs through:

  1. Register with FortiCare and enable SSO (or import an entitlement file in a closed network)
  2. Set the hostname
  3. Change the admin password
  4. Optionally upgrade firmware

The wizard stops appearing once you complete it.

Administrator Accounts & Profiles

Two profile types:

  • System Admin — full granularity. Choose Read-Write, Read-Only, or None for each module (System Settings, FortiGuard, Device Manager, etc.).
  • Restricted Admin — scoped to a small set of security profiles per ADOM (web filter, IPS sensors, application sensors). Useful for delegating tuning without giving structural access.

Access can be controlled by combining ADOMs (which devices an admin can see), the profile (what they can do), and trusted hosts (which IPs they can log in from). External authentication is supported via LDAP, RADIUS, TACACS+, and PKI, with optional 2FA via FortiAuthenticator / FortiToken Cloud.

Concurrent Admins — Workspace & Workflow

By default workspace-mode is disabled: multiple admins can edit the same ADOM at the same time and step on each other. The fix is a locking mode.

ModeCLI valueWhat it adds
DisableddisabledNo locks (default).
Workspace (ALL ADOMs)normalLocking on every ADOM.
Workspace (Per-ADOM)per-adomLocking only on chosen ADOMs.
Workflow (ALL ADOMs)workflowLocking + an approval pipeline before installation.

Locks can be taken at multiple levels: ADOM, device, policy package, object, or — with Per-Policy Lock — individual policies. Lock icons in the GUI: open (unlocked), green closed (locked by you), red closed (locked by another admin).

Workflow mode is the right choice when changes must be reviewed and approved before being applied — e.g. environments with change-control or segregation-of-duties requirements.

ADOM Best Practices

  • Group devices by FortiOS firmware version first — command syntax varies between versions, and one-ADOM-per-firmware avoids script and template surprises.
  • Other axes to layer on: assigned admin team, geographic region, customer (MSSP), or environment (prod / dev / test).
  • An ADOM’s version determines which device firmware versions it can manage — check the support matrix before planning upgrades.
  • Upgrade managed devices first, then upgrade the ADOM (or, if you upgrade the ADOM first, install pending changes and check the install preview after).
  • FortiManager firmware upgrades do not auto-upgrade ADOMs. The ADOM upgrade is a separate manual step.
  • Many stale ADOM revisions slow upgrades — prune within retention policy before kicking one off.
  • Moving a device to a new ADOM doesn’t carry its policy package or shared objects — re-import them.

Backup, Restore, Migration, Offline Mode

The backup contains everything except logs, FortiGuard cache, and saved firmware images:

  • All managed devices
  • Global database
  • Flash configuration

Backups can run on demand or on a schedule, over FTP / SCP / SFTP, with optional password encryption. Restores reboot FortiManager.

Restore options

  • Overwrite current IP, routing, and HA — on by default; turn off if you’re restoring on the same box and want to keep its current network identity.
  • Restore in offline mode — always on; FortiManager comes back up with FGFM disabled so it can’t collide with another active manager during the restore.

Migrating to a different model

execute migrate all-settings < ftp | scp | sftp > <server> <filepath> <user> <password> [cryptpasswd]

Backs up on model A, restores on model B. Useful for hardware swaps or moving up to a larger appliance. FortiGuard databases (AV, IPS, etc.) aren’t inside the config and need to be re-pulled.

Offline mode

Disabled by default. Toggling it on disables the FGFM protocol so FortiManager won’t talk to managed devices. Used for safe troubleshooting, lab cloning, or restoring a backup on a second FortiManager without it fighting the live one.

Reset

execute reset all-settings        # full factory reset
execute reset all-except-ip       # reset, keep IP/routing
execute format disk ...           # erase databases & logs, repartition

Provisioning Templates & Mass Provisioning

Templates are how FortiManager actually does mass provisioning — define a setting once (CLI, system, IPsec, SD-WAN, BGP, certificate, etc.) and assign it to many FortiGates. The template becomes the source of truth for whatever it covers, so manual edits to the device DB outside the template will be overwritten on the next push.

Normalized (dynamic) interfaces are the policy-layer abstraction for interfaces — they let you write a policy once and map LAN / WAN / etc. to whichever physical interface a given FortiGate model actually has. Each device needs a per-device mapping (or a per-platform mapping for a model family). A missing mapping triggers Dynamic interface mapping undefined install errors.

Revision History & ADOM Revisions

Two different revision concepts — easy to confuse:

Device revision historyADOM revision
ScopeOne device’s device-level DBAn ADOM’s policy packages & objects
Created whenFortiManager installs to the device, or auto-retrieves out-of-band changesManually, or via the Install Wizard’s Policy & Package Settings
Reverting itRewrites the device-level DB on FortiManager (no policy-layer effect)Rewrites policy packages and objects in the ADOM

Reverting a device revision does not reach the FortiGate on its own — you still need to install for the change to land.

Two Databases — ADOM-level vs Device-level

Every managed FortiGate’s configuration is split across two databases on FortiManager. Knowing which side something lives on tells you where to make the change and how to push it.

ADOM-level (policy / objects)Device-level
Firewall policies, security profiles (AV, IPS, web filter, app control, DLP), address & service objects, schedules, normalized interfaces Interfaces, routing (static, BGP, OSPF), DNS, SNMP, HA, DHCP, NSX-T service template — anything tied to a specific device

Scripting Targets

FortiManager scripts (CLI or Tcl) can run against three different targets:

  • Policy package database — accepts policy / object syntax only.
  • Device database — accepts device-level CLI like config system interface or config router bgp.
  • Remote FortiGate (via CLI) — runs the script live against the FortiGate’s real CLI.

Pointing a script at the wrong target produces object unrecognized errors. Device-level commands fail against the policy DB, and vice versa.

FortiManager Behind NAT

When FortiManager sits behind a NAT firewall, the FortiGates it manages can’t reach its internal IP — they only see the public/VIP. The mgmt-addr setting tells FortiManager which address to advertise to managed FortiGates: 

config system admin setting
    set mgmt-addr 100.65.0.120   # the public/NATed IP
end

During discovery, FortiGate is told to talk to this single address only. The internal IP is never exposed. If both ends are behind NAT, both sides need their own public address configuration so the FGFM tunnel can be established and re-established when it drops.
